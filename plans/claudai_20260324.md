# FactoryCraft 1.1 — Backend Architecture Plan

## Context

Building a Builderment-style factory automation game from scratch. The backend is a Go REST API deployed on Google Cloud Run (free tier), using Firestore as the database. Frontend is TBD — backend is designed to be frontend-agnostic. GCP exam prep is a secondary goal, so idiomatic GCP service usage is prioritized. Full game loop in scope: auth, worlds, map generation, buildings, conveyor automation, crafting, and research.

## Firestore Data Model

```
players/{playerID}
  id, email, displayName, createdAt, researchPoints

  worlds/{worldID}
    id, name, seed, createdAt, lastSimulatedAt, width, height

    resourceNodes/{nodeID}
      type: "forest"|"iron_mine"|"copper_mine"|"stone_mine"|"gold_mine"|"coal_mine"|"uranium"|"wolframite"|"oil_well"|"silicon_mine"
      x, y
      # Nodes are INFINITE — no depletion. richness removed.

    buildings/{buildingID}
      type: "extractor"|"conveyor"|"splitter"|"merger"|"factory"|"research_lab"
      x, y, rotation, recipeID, linkedNodeID
      inputSlots: map<itemID, quantity>
      outputSlots: map<itemID, quantity>
      connections: []buildingID  (ordered output connections)
      isActive, lastTickAt

    inventory/state
      items: map<itemID, quantity>
      coins: int64                          # world-level coin balance
      researchPoints: int64                 # world-level research point balance
      totalDelivered: map<itemID, int64>    # cumulative items ever sold at Research Lab (never decreases)

    research/state                          # per-world research progress
      unlockedNodes: []researchNodeID
      updatedAt: timestamp

research/{nodeID}                   (static, loaded once at startup from research.json)
  name, description
  deliveryRequirements: [{itemID, quantity}]  # items that must have been DELIVERED (cumulative) to Research Lab
  researchPointCost: int                      # research points consumed on unlock
  coinCost: int                               # coins consumed on unlock
  prerequisites: []nodeID                     # other nodes that must be unlocked first
  unlocks: []string                           # buildingTypes or recipeIDs unlocked
```

**Key design notes:**
- `lastSimulatedAt` on world document drives lazy time-delta simulation
- `inputSlots`/`outputSlots` as maps avoids per-building buffer subcollections → cheap reads
- Research is **per world** — `research/state` subcollection under each world
- `totalDelivered` is append-only (never decreases) — tracks cumulative research station deliveries
- Static research tree loaded from `research.json` at startup; only `research/state` and `inventory/state` are read per-request
- Each world load = ~4-10 Firestore reads (well within 50k/day free tier)

## Simulation Approach: Lazy Time-Delta on Read

Cloud Run scales to zero → no always-on tick loop. Instead:

1. On `GET /map` or `POST /tick`: read `lastSimulatedAt`, compute `deltaSeconds = now - lastSimulatedAt`
2. If `deltaSeconds < 1s` → return current state (skip write)
3. Cap delta at `maxOfflineSeconds` (e.g. 8 hours) to prevent runaway accumulation
4. Run simulation engine (pure function, no Firestore calls inside):
   - Extractors: `yields = floor(rate × delta)`, update node `depletionFactor`
   - Conveyors (topological order): move items between building buffers
   - Splitters/Mergers: apply routing rules
   - Factories: `completedCycles = floor(delta / recipeDuration)`, consume inputs, produce outputs
   - Research Labs: accumulate research points
5. Write all mutations in a **Firestore transaction** (read-compute-write as one atomic op)

**Simulation engine types:**
```go
type WorldState struct {
    World         WorldDoc
    ResourceNodes map[string]ResourceNodeDoc
    Buildings     map[string]BuildingDoc
    Inventory     map[string]int64
}

type SimulationResult struct {
    MutatedBuildings  map[string]BuildingDoc
    MutatedNodes      map[string]ResourceNodeDoc
    InventoryDelta    map[string]int64
    NewLastSimulated  time.Time
}
```

The engine is a pure function — zero Firestore dependency — enabling 100% unit test coverage.

**Resource node types (10 total):** `forest`, `iron_mine`, `copper_mine`, `stone_mine`, `gold_mine`, `coal_mine`, `uranium`, `wolframite`, `oil_well`, `silicon_mine`

## Extractor Mechanics

### Placement Rules
- Must be placed on the exact `x,y` of a resource node — validated at `POST /buildings`
- `linkedNodeID` on the building document stores which node it's on
- One extractor per node (no stacking)
- `rotation` (0/90/180/270°) sets the single output direction

### Base Extraction Rates (items/minute, defined in `recipes.json`)

| Resource | Output Item | Base Rate |
|---|---|---|
| Forest | Wood | 120/min |
| Iron Mine | Iron Ore | 60/min |
| Copper Mine | Copper Ore | 60/min |
| Stone Mine | Stone | 60/min |
| Gold Mine | Gold Ore | 30/min |
| Coal Mine | Coal | 60/min |
| Oil Well | Crude Oil | 30/min |
| Silicon Mine | Raw Silicon | 30/min |
| Uranium | Uranium Ore | 10/min |
| Wolframite | Wolframite Ore | 10/min |

Rates are data-driven (in `recipes.json`) — tunable without recompiling.

### Upgrade Multipliers (via Research)
Research nodes can unlock extractor tier upgrades that apply a multiplier to base rate:
- Tier 1 (default): ×1
- Tier 2 (research unlock): ×2
- Tier 3 (research unlock): ×4

The multiplier is stored on the building document as `extractorTier: int` (1/2/3).
Effective rate = `baseRate × tierMultiplier`.

### Output Buffer
- Extractor has an `outputSlots` map with one item type and a max capacity (e.g. 50 items)
- If the output buffer is full and no belt is connected (or belt is backed up), production pauses
- During simulation tick: `produced = floor(effectiveRate/60 × deltaSeconds)`, add to buffer up to cap

### Simulation Engine Logic (extractor step)
```
for each active Extractor:
  baseRate  = recipe.Registry.ExtractionRate[building.linkedNodeType]
  tierMult  = tierMultipliers[building.extractorTier]
  produced  = floor((baseRate * tierMult / 60) * deltaSeconds)
  current   = building.outputSlots[itemID]
  building.outputSlots[itemID] = min(current + produced, outputBufferCap)
```

## Economy

### Two Currencies

| Currency | Earned by | Spent on |
|---|---|---|
| **Coins** | Delivering items to Research Lab | Placing buildings, unlocking research |
| **Research Points** | Delivering items to Research Lab | Unlocking research nodes |

Both currencies are stored on `inventory/state` per world (not per player — each world has its own economy).

### Item Pricing (in `items.json`)
Each item definition carries:
```json
{
  "id": "iron_ingot",
  "name": "Iron Ingot",
  "sellPrice": 10,          // coins earned when delivered to Research Lab
  "researchValue": 2,       // research points earned when delivered to Research Lab
  "buyPrice": 8             // used to calculate building placement cost
}
```
Raw resources have low sell prices; advanced products have high sell prices — incentivizing deep production chains.

### Earning Coins & Research Points (Research Lab)
When items arrive in the Research Lab's input buffer during a simulation tick:
```
for each itemID, quantity in researchLab.inputSlots:
  coinsEarned              += items[itemID].sellPrice     * quantity
  researchPointsEarned     += items[itemID].researchValue * quantity
  inventory.totalDelivered[itemID] += quantity            // cumulative, never resets
  researchLab.inputSlots[itemID] = 0                      // items consumed instantly

inventory.coins          += coinsEarned
inventory.researchPoints += researchPointsEarned
```
The Research Lab has **no output buffer** — items are consumed immediately on arrival.
`totalDelivered` is the unlock gate for research nodes — it only ever increases.

### Building Placement Cost
Each building type has an explicit `placementCost` defined in `buildings.json`:
```json
{
  "type": "assembler",
  "placementCost": {
    "coins": 50,
    "items": [
      { "itemID": "iron_plate", "quantity": 5 },
      { "itemID": "circuit_board", "quantity": 2 }
    ]
  }
}
```
At `POST /worlds/{worldID}/buildings`:
1. Check `inventory.coins >= placementCost.coins`
2. Check `inventory.items` has enough of each required item
3. If valid: deduct coins + items from inventory, create building document — all in one Firestore transaction

Starter buildings (Extractor, Conveyor, basic Research Lab) have low coin costs and no item requirements, so early-game bootstrapping is possible.

### Starter Coins
New worlds are created with a configurable `startingCoins` balance (default: 500 coins, defined in server config). This allows the player to immediately place extractors and belts before any items have been produced.

### Research Unlock Requirements
A research node can only be unlocked when ALL of these are satisfied:
1. **Delivery requirements met**: `inventory.totalDelivered[itemID] >= node.deliveryRequirements[itemID]` for each required item
2. **Enough research points**: `inventory.researchPoints >= node.researchPointCost`
3. **Enough coins**: `inventory.coins >= node.coinCost`
4. **Prerequisites unlocked**: all `node.prerequisites` present in `research/state.unlockedNodes`

`GET /worlds/{worldID}/research` returns the full tree with per-node progress so the frontend can show:
> Wood Processing: 87/100 Wood delivered · 30/50 RP · prerequisites met ✓

```
POST /worlds/{worldID}/research/{nodeID}/unlock
  Firestore transaction:
    read inventory/state + research/state
    validate all 4 conditions above
    deduct researchPoints + coins from inventory/state
    append nodeID to research/state.unlockedNodes
    write both documents atomically
```

## Factory Mechanics

### Factory Building Types
Multiple specialized buildings, each handling a tier of recipes. Unlocked via research.

| Building | Purpose | Example Recipes |
|---|---|---|
| **Smelter** | Ore → refined material | Iron Ore → Iron Ingot, Copper Ore → Copper Ingot |
| **Assembler** | Basic crafting | Iron Ingot → Iron Plate, Wood → Wood Plank, Iron Plate + Copper Wire → Circuit Board |
| **Advanced Assembler** | Complex multi-input crafting | Circuit Board + Iron Plate + Copper Wire → Advanced Circuit |
| **Chemical Plant** | Oil/chemical processing | Crude Oil → Petroleum + Lubricant *(byproduct)* |
| **Research Lab** | Consumes items → research points | Various items → research points (no physical output) |

### Recipe Assignment
- Player manually selects a recipe from the building's available list (filtered by factory type + research unlocks)
- Stored as `recipeID` on the building document
- Factory does nothing until a recipe is assigned

### Recipe Structure (`recipes.json`)
```json
{
  "id": "oil_refining",
  "name": "Oil Refining",
  "factoryType": "chemical_plant",
  "craftingTimeSec": 5.0,
  "inputs": [
    { "itemID": "crude_oil", "quantity": 3 }
  ],
  "outputs": [
    { "itemID": "petroleum",  "quantity": 2, "isPrimary": true  },
    { "itemID": "lubricant",  "quantity": 1, "isPrimary": false }
  ]
}
```

- `inputs`: 1–3 item types
- `outputs`: 1 primary + 0 or 1 byproduct
- `craftingTimeSec`: time per production cycle

### Input/Output Buffers
- `inputSlots: map<itemID, quantity>` — holds items delivered by incoming belts
- `outputSlots: map<itemID, quantity>` — holds both primary product AND byproduct
- **Single output connection** — all items in `outputSlots` flow out onto one belt together (mixed items)
- Player uses a Splitter after the factory to separate primary from byproduct if needed
- Buffer cap: e.g. 50 items per slot; production pauses if output buffer is full

### Simulation Engine Logic (factory step)
```
for each active Factory with a recipeID:
  recipe = recipe.Registry.GetRecipe(building.recipeID)

  // How many full cycles can we complete?
  cyclesCap = floor(deltaSeconds / recipe.craftingTimeSec)
  for each input in recipe.inputs:
    cyclesCap = min(cyclesCap, floor(building.inputSlots[input.itemID] / input.quantity))

  if cyclesCap == 0: continue  // not enough inputs or time

  // Check output buffer has space
  for each output in recipe.outputs:
    space = outputBufferCap - building.outputSlots[output.itemID]
    cyclesCap = min(cyclesCap, floor(space / output.quantity))

  if cyclesCap == 0: continue  // output buffer full

  // Apply production
  for each input  in recipe.inputs:  building.inputSlots[input.itemID]   -= input.quantity  * cyclesCap
  for each output in recipe.outputs: building.outputSlots[output.itemID] += output.quantity * cyclesCap
```

### Research Lab (special case)
- No physical output — consuming items generates `researchPoints` on the player document
- Configured as a recipe with `outputs: [{ itemID: "research_points", quantity: N }]`
- During simulation tick, research points accumulate in `playerResearch.pendingPoints`; written to player doc in the same Firestore batch

## Conveyor Belt Mechanics

### Design Summary
- **Mixed items**: a belt carries any item types simultaneously
- **Instant transfer per tick**: no per-tile item queues; items move directly from source output buffer to destination input buffer during each simulation tick
- **Explicit Splitter/Merger buildings**: required at any junction — a plain belt tile has exactly 1 input and 1 output
- **Global belt speed tier**: unlocked via research; applies to ALL belts in ALL worlds for that player simultaneously

### Belt Tile on the Map
Each belt tile is a `building` document of type `"conveyor"`:
- `x, y` — grid position
- `rotation` — visual direction (0/90/180/270°), also determines which grid neighbor is the output
- `connections: [nextBuildingID]` — the single building this belt outputs to (next belt tile or a machine)
- **No input/output buffer** on belt tiles — they are routing connectors only, not storage

### Global Belt Speed Tiers (Research Unlock)
Stored as `beltTier: int` on the `playerResearch/{playerID}` document. Applies to every belt in every world.

| Tier | Throughput | Research Required |
|---|---|---|
| 1 (default) | 60 items/min | none |
| 2 | 120 items/min | Belt Speed I |
| 3 | 240 items/min | Belt Speed II |

### Simulation: How Belt Transfer Works
Belt tiles are **skipped** during simulation. The engine traverses the connection graph to find the next *real* building (extractor, factory, splitter, merger, or inventory sink) at the end of a belt chain. Transfer happens directly between the source output buffer and the destination input buffer.

```
For each building with a non-empty outputSlot AND a connection:
  Follow connection chain, skipping conveyor tiles, to reach destination building
  throughputCap = (playerBeltSpeed / 60) × deltaSeconds   // items this tick
  for each itemType in source.outputSlots:
    transferable = min(source.outputSlots[itemType], throughputCap, destinationInputCapacity)
    source.outputSlots[itemType]      -= transferable
    destination.inputSlots[itemType]  += transferable
```

Items of all types share the throughput cap on a single belt chain (first-come-first-served by item iteration order in the map).

### Splitter Building
- 1 input, up to 3 outputs (set via `connections` array, max length 3)
- Distributes items **round-robin** across active outputs per tick
- Each output gets `floor(throughputCap / numOutputs)` items per item type
- If an output is backed up (destination input buffer full), that output is skipped and its share redistributed to remaining outputs

### Merger Building
- Up to 3 inputs, 1 output
- Pulls items from all connected input sources each tick, combines into single output connection
- Total throughput still capped by global belt speed tier
- No filtering — all item types pass through

### Connection Validation at Placement
When a belt/splitter/merger connection is created (`POST /buildings/{id}/connect`):
- Validate no cycle is formed (DFS cycle detection on the building graph)
- Validate belt tile at `x,y` neighbors match rotation direction
- Splitter: reject if already has 3 connections
- Merger: reject if target already has 3 incoming connections

## Dev Map Visualization

Two debug endpoints served from `internal/debug/`, disabled in production via env flag `DEBUG_ROUTES=false`:

### 1. SVG Snapshot — `GET /worlds/{worldID}/debug/map.svg`
- Go generates SVG inline using `encoding/xml` or string templating — no library needed
- Each resource node: colored square + 1-letter label (F=Forest, I=Iron, C=Copper, etc.)
- Each building: different color + directional arrow based on `rotation`
- Belt connections: thin lines between connected buildings
- Open directly in browser or embed in docs; useful for quick snapshots

### 2. Interactive HTML Canvas — `GET /debug/map/{worldID}`
- Go embeds `static/debug_map.html` using `//go:embed` (stdlib, no file serving setup needed)
- The HTML page fetches `GET /api/v1/worlds/{worldID}/map` using the world ID from the URL
- Vanilla JS draws tiles on `<canvas>`: resource nodes colored by type, buildings with rotation indicator
- Click a tile → shows JSON of that entity in a sidebar
- Mouse wheel zoom + drag to pan
- Uses a `?token=` query param for the Firebase auth token in dev (no login UI needed)
- ~150 lines of vanilla JS, zero npm/bundler required
